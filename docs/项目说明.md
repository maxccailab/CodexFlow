# CodexFlow 项目说明

本文档用于解释当前项目的核心运行逻辑、配置参数、运行过程中会生成的文件，以及这些文件分别承担什么作用。

## 1. 项目定位

当前项目是一个用 Go 编写的本地编排器。它本身不直接实现复杂智能体能力，而是负责：

- 读取一个 JSON 任务配置文件
- 根据配置选择当前角色
- 调用本地已经安装的 `codex` 命令行程序
- 要求 `codex` 按固定 JSON 结构返回结果
- 根据返回结果决定是否继续、结束或阻塞
- 保存运行状态、角色输出和历史记录

可以把它理解为：

1. Go 程序负责流程控制
2. 本地 `codex` 负责每一轮具体执行和判断
3. 配置文件负责描述任务目标和角色关系

## 2. 程序入口

入口文件是：

- [main.go](/Users/cc/opensource/CodexFlow/cmd/codexflow/main.go)

启动方式示例：

```bash
go run ./cmd/codexflow --config ./examples/feature-a.json --dir .
```

程序启动后会先做以下事情：

1. 解析命令行参数
2. 校验工作目录是否存在
3. 读取并校验任务配置文件
4. 检查本地 `codex` 命令是否可用
5. 创建运行状态目录
6. 进入按角色串行执行的主循环

## 3. 命令行参数

当前程序支持两个命令行参数。

### `--config`

作用：

- 指定任务配置 JSON 文件路径

特点：

- 必填
- 支持相对路径
- 支持绝对路径

示例：

```bash
--config ./feature-a.json
```

如果不传，程序会直接报错退出。

### `--dir`

作用：

- 指定工作目录

这个目录会被用于：

- 作为 `codex` 的 `--cd` 目录
- 作为运行状态目录 `.codexflow/` 的根位置
- 作为提示词中的“工作目录”

特点：

- 可选，默认是 `.`
- 相对路径会相对于程序启动时的当前目录解析
- 必须是真实存在的目录

示例：

```bash
--dir .
--dir /Users/cc/opensource/CodexFlow
```

## 4. 配置文件结构

任务配置文件对应的 Go 结构体是 `TaskConfig` 和 `RoleConfig`。

### 顶层字段

#### `task_id`

作用：

- 任务唯一标识
- 用于决定状态目录名称

例如：

```json
"task_id": "feature-a"
```

#### `goal`

作用：

- 描述整个任务的总目标
- 会被注入到每一轮提示词中

例如：

```json
"goal": "完成一个最小闭环"
```

#### `initial_role`

作用：

- 指定第一轮从哪个角色开始执行

要求：

- 必须在 `roles` 中存在

例如：

```json
"initial_role": "设计者"
```

#### `max_turns`

作用：

- 限制最大轮次数，避免无限循环

例如：

```json
"max_turns": 12
```

程序达到这个上限后会结束，并打印：

```text
流程结束：已达到最大轮次 N。
```

#### `roles`

作用：

- 定义所有角色

要求：

- 至少有一个角色
- 角色名不能重复

### 角色字段

每个角色对应一个 `RoleConfig`。

#### `name`

作用：

- 角色名称
- 运行时通过它定位当前角色和下一角色

#### `description`

作用：

- 角色描述
- 会写入提示词，告诉模型当前角色是什么

#### `instructions`

作用：

- 角色职责说明
- 会写入提示词，约束本轮行为

#### `prompt`

作用：

- 当前角色的默认提示词
- 每轮都会以该角色自己的 `prompt` 作为本轮指令

#### `allowed_next_roles`

作用：

- 列出这个角色允许交接到哪些角色

特点：

- 可以为空
- 如果为空，模型就不应该再继续交接
- 如果有多个候选，当前角色需要在这些候选中选择 `next_role`

## 5. 运行逻辑

当前主流程在 `run()` 中。

### 第一步：初始化

程序会：

1. 解析 `--dir`
2. 校验目录存在
3. 读取 `--config`
4. 校验配置合法性
5. 创建状态目录
6. 检查本地 `codex` 是否在 `PATH` 中
7. 写出 JSON Schema 文件

### 第二步：恢复运行状态

程序会尝试读取：

```text
<dir>/.codexflow/<task_id>/runtime-state.json
```

如果这个文件存在，程序会恢复：

- 当前已经执行到第几轮
- 当前活跃角色是谁
- 最近一次交接内容是什么

如果这个文件不存在，则从配置中的 `initial_role` 和第 0 轮开始。

### 第三步：生成当前角色提示词

每轮执行前，程序会调用 `buildRolePrompt()` 生成中文提示词。

提示词内容主要包括：

- 任务目标
- 工作目录
- 当前轮次
- 当前角色
- 角色描述
- 角色职责
- 允许交接到哪些角色
- 最近一次交接摘要
- 最近一次交接事项
- 当前这一轮的指令
- 输出格式要求

提示词中还明确要求：

- 必须按 JSON Schema 输出
- `status` 只能是 `continue`、`blocked`、`complete`
- 如果继续，必须返回合法的 `next_role`
- 如果完成或阻塞，`next_role` 必须为空

### 第四步：调用本地 `codex`

程序内部调用的是本地命令行：

```bash
codex exec ...
```

或者在已有角色会话的情况下调用：

```bash
codex exec resume <session_id> ...
```

调用时固定带上：

```bash
--color never
--dangerously-bypass-approvals-and-sandbox
--cd <dir>
--output-schema <schemaPath>
-o <outputFile>
```

含义：

- `--cd <dir>`：让 `codex` 在指定工作目录中运行
- `--output-schema`：强制模型按指定 JSON Schema 输出
- `-o <outputFile>`：把最终结构化结果写入文件

### 第五步：读取和校验角色输出

程序读取角色的结构化输出文件，并解析成 `Handoff` 结构。

关键字段包括：

- `status`
- `reply_to_user`
- `handoff_summary`
- `handoff_items`
- `next_role`
- `completion_reason`
- `completion_confidence`

程序会执行校验：

- `status` 必须合法
- `completion_confidence` 必须在 0 到 1 之间
- `reply_to_user` 不能为空
- `handoff_summary` 不能为空

如果 `status=continue`：

- `next_role` 不能为空
- `next_role` 必须属于 `allowed_next_roles`

如果 `status=blocked`：

- `completion_reason` 不能为空

如果 `status=complete` 或 `status=blocked`：

- `next_role` 必须为空

### 第六步：打印终端信息

程序会把当前轮的结果打印到终端。

#### 当 `status=continue`

终端会显示：

- `reply_to_user`
- `下一角色`

#### 当 `status=blocked` 或 `status=complete`

终端会显示：

- `reply_to_user`
- `结束原因`

### 第七步：写入历史和状态

每一轮执行后，程序会做两件事：

1. 把这一轮摘要追加写入 `history.jsonl`
2. 把最新运行状态写入 `runtime-state.json`

如果状态是：

- `continue`：切换到 `next_role`，继续下一轮
- `blocked`：结束流程
- `complete`：结束流程

## 6. 角色会话机制

每个角色都有自己的独立会话文件。

这意味着：

- “设计者” 有自己的 `session id`
- “审核者” 有自己的 `session id`
- 下次再次切换到同一个角色时，会尝试恢复这个角色自己的上下文

对应逻辑在：

- `runCodexCommand()`
- `readSessionID()`
- `storeSessionID()`
- `shouldResetSession()`

如果角色会话失效，例如 `codex` 返回类似：

- `session not found`
- `invalid session`

程序会删除旧的 session 文件，然后重新开一个新会话。

## 7. 运行过程中会产生哪些文件

以下文件都会出现在：

```text
<dir>/.codexflow/<task_id>/
```

假设：

- `--dir` 是项目根目录
- `task_id` 是 `feature-a`

那么目录大致是：

```text
.codexflow/
└── feature-a/
    ├── role-output.schema.json
    ├── runtime-state.json
    ├── history.jsonl
    ├── 设计者.session
    ├── 设计者.last-output.json
    ├── 设计者.last-raw.txt
    ├── 审核者.session
    ├── 审核者.last-output.json
    └── 审核者.last-raw.txt
```

下面分别解释。

### `role-output.schema.json`

作用：

- 给 `codex` 作为输出格式约束使用

程序会在调用 `codex` 时通过 `--output-schema` 指向这个文件。

它本质上是“模型输出契约”，确保 `codex` 必须返回程序可以解析的 JSON 结构。

### `runtime-state.json`

作用：

- 保存整个编排器的运行状态

包含：

- 当前轮次
- 当前活跃角色
- 任务目标
- 工作目录
- 最近一次 handoff

用途：

- 程序重启后继续执行

### `history.jsonl`

作用：

- 保存每一轮的摘要历史

格式：

- 一行一条 JSON

每条记录包含：

- 第几轮
- 当前角色
- 当前状态
- 下一角色
- 本轮摘要
- 结束原因
- 结构化输出文件路径
- 原始输出文件路径

用途：

- 回放整条执行链
- 排查流程为何进入某个角色
- 追踪任务如何一步步推进

### `<角色名>.session`

作用：

- 保存这个角色对应的 `codex` 会话 ID

用途：

- 下次再次执行这个角色时，优先用 `codex exec resume` 恢复这个角色的上下文

### `<角色名>.last-output.json`

作用：

- 保存这个角色最近一次结构化输出

内容：

- 就是符合 JSON Schema 的 handoff 结果

用途：

- 供 Go 程序解析
- 供人工查看该角色最后一次返回了什么

### `<角色名>.last-raw.txt`

作用：

- 保存这个角色最近一次 `codex` 的原始 CLI 输出

用途：

- 当结构化 JSON 解析失败时排错
- 当输出校验失败时回看模型到底输出了什么
- 辅助判断是 prompt 问题、schema 问题，还是模型执行过程的问题

## 8. 当前终端输出代表什么

程序本身会把两类内容打印到终端。

### 一类是 `codex` 自己的原始输出

这是因为程序执行 `codex` 后，会直接把 `CombinedOutput()` 打印出来。

这部分可能包含：

- `session id`
- `codex` 的过程输出
- 其他运行日志

### 另一类是编排器整理后的轮次摘要

例如：

```text
[第 3 轮][审核者]
已完成当前检查
下一角色: 设计者
```

或者：

```text
[第 5 轮][测试者]
已确认当前流程存在阻塞
结束原因: 缺少必要测试数据
```

这是项目真正面向使用者的摘要输出。

## 9. 程序的关键设计点

### 设计点 1：流程控制和模型执行分离

Go 程序不负责“智能判断怎么改代码”，只负责：

- 配置解析
- 状态管理
- 调用本地 `codex`
- 校验输出
- 决定流程往哪里走

### 设计点 2：每个角色有自己的会话

这样每个角色切回来的时候，能够保留自己的上下文连续性。

### 设计点 3：编排状态和角色状态分开保存

- 编排状态：`runtime-state.json`
- 角色状态：`<role>.session`

这样既能恢复总流程，也能恢复每个角色自己的会话。

### 设计点 4：结构化输出优先

程序依赖 JSON Schema 和结构化 JSON，而不是解析自然语言。

这是当前程序稳定运行的基础。

## 10. 当前项目的局限

虽然当前已经可用，但仍有几个边界：

- 不支持并行角色执行
- 不支持一个角色同时交接给多个角色
- 不支持复杂分支合流
- 不支持自动清理旧状态
- 不支持通过命令行直接重置某个任务状态

目前它更适合：

- 小规模单任务、多角色、串行编排

## 11. 建议如何理解这个项目

如果用一句话概括：

这个项目是一个“用 Go 写的本地多角色串行调度器”，它通过 JSON 配置和本地 `codex` CLI，把一个任务拆成多轮角色执行，并把全过程状态化、可恢复、可追踪。
